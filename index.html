CONTEXT ENGINEERING SERIES — PART 2

Teaching Claude to Connect

Part 1 gave Claude memory. But memory without hands is just a journal. MCP servers give Claude hands — direct access to your CRM, your code, your docs, and your channels.

9 servers. 120+ tools. One protocol. And a case study where corporate IT said "no Connected App" — so we built around it.

The Problem

Smart but Isolated

After Part 1, Claude remembers everything — your corrections, your architecture decisions, your product naming rules. But when you say "pull up the Roblox account," it can't. When you say "check what's in Slack," it can't. When you say "look at that PR," it can't.

🔒

No CRM Access

"What does Roblox own?" You have to manually look it up in Salesforce, navigate across multiple account variants, screenshot the asset page, paste it into the conversation, and hope Claude interprets the formatting correctly.

🔌

No Code Context

"Review that PR." You copy the diff, paste it, lose half to context window limits. Claude reviews a fragment without understanding the repo.

🔇

No Communication

"Send the follow-up to the customer." Not possible. Claude drafts the email. You copy it. Open Slack. Paste it. Adjust the formatting. Send.

The tax is invisible. Every time you Alt-Tab to look something up and paste it back, you're burning 30-60 seconds and cluttering the context window with raw, unstructured data. Do that 20 times in a session and you've lost 10-20 minutes — plus used 30% of your context window on copy-paste artifacts instead of actual work.

The Protocol

MCP in 30 Seconds

Model Context Protocol is an open standard from Anthropic. It lets Claude call external tools — APIs, databases, CLIs, browsers — through a standardized interface. Think of it as USB-C for AI: one protocol, any capability.

P

Protocol

JSON-RPC standard. Any host (Claude Code, Cursor, Windsurf) speaks the same language to any server.

S

Server

A lightweight wrapper around an API, CLI, or service. Runs locally or remote. Exposes tools Claude can call.

T

Tool

A single capability: "search Slack," "read a PR," "query CRM." Claude decides when to call it. You approve or auto-allow.

Case Study

The Impossible Connection

Salesforce's corporate CRM (Org62) holds everything — pipeline, contacts, contracts, installed products, consumption data. It's the single source of truth for every deal.

We needed it inside Claude. Corporate IT said no.

The Constraint

×
No Connected App in Org62

Can't create one. Requires internal IT approval that doesn't exist for individual SE tooling. This kills JWT Bearer Flow dead.

×
No API credentials

No client_id, no client_secret, no certificate. The standard OAuth patterns are all off the table.

×
No direct middleware connection

Our Cloud Run middleware can't query Org62. No JWT, no OAuth, no cert — it's a hard wall between production infra and corporate CRM.

The Solution: sf CLI + MCP

✓
Salesforce CLI authenticated locally

sf org login web --alias org62 — browser-based SSO login. Uses your existing corporate credentials. No app registration needed.

✓
MCP server wraps sf CLI commands

A custom Node.js MCP server that calls sf data query --query "SELECT..." --target-org org62 under the hood. Claude calls the tool, the server runs the CLI.

✓
Cache-forward architecture for production

MCP gathers data → pushes to middleware cache (Redis 4h / Cloud SQL 24h) → all production entry points (LWC, Agentforce, Slack) read from cache. Fix in one place, available everywhere.

New Addition

Slack MCP — The Second Data Plane

The newest server in the stack — and it turned out to be more important than expected. Claude can now search Slack, read channels and threads, send messages, and create canvases. But the real value is what it surfaces: the unstructured human signal that no CRM captures.

What It Unlocks

  • • Search across channels — find that conversation about a deal strategy from last week without scrolling
  • • Read thread context — Claude can follow a thread to understand the full discussion before drafting a response
  • • Send messages — draft and post directly. Meeting follow-ups, deal updates, team announcements
  • • Create canvases — publish long-form content like this very post series, formatted and ready to share

SE Workflow: Post-Meeting Follow-Up

me: Send the meeting notes to #acme-deal and DM Sarah the action items
  // Claude calls slack_search_users("Sarah")
  // Claude calls slack_search_channels("acme-deal")
  // Claude calls slack_send_message (channel)
  // Claude calls slack_send_message (DM)
claude: Done. Posted summary to #acme-deal and sent Sarah the 3 action items with dates.
Architectural Insight

Two Data Planes. One Account.

After authenticating to corporate Slack, I had a question: does Slack MCP make the Meridian CRM tools redundant? If I can search Slack for "Roblox," why query Org62 separately?

I tested it. The answer is a definitive no — and the reason reveals something important about how MCP servers compose.

💬

Slack MCP

Unstructured. Conversational. Human signal.
What it returns for "Roblox"
  • • Team dynamics — who's working the deal, who's the AE, who's looped in on technical
  • • Competitive intel — mentions of competing platforms, win/loss context from deal channels
  • • Customer sentiment — how the customer reacted in meetings, what concerns were raised
  • • Risk signals — executive changes, project delays, escalations flagged by the team
The story. The context. The human judgment layer.
📊

Meridian MCP (Org62)

Structured. Quantitative. System of record.
What it returns for "Roblox"
  • • Installed products — exact SKUs, quantities, per-unit pricing, discount percentages, contract periods
  • • Deal history — every won, lost, and open opportunity with amounts and close dates
  • • Account variants — multi-org customers appear as separate CRM accounts (core, Slack, Tableau)
  • • Consumption health — CCR utilization, contract values, renewal timelines
The numbers. The contracts. The single source of truth.

The Architectural Insight

MCP servers don't need to talk to each other. They don't even need to know each other exists. Claude is the integration layer — it decides which servers to call and synthesizes the results.

Think of it like Unix pipes. Each MCP server is a specialized command — grep, sort, awk. The LLM is the pipe operator | that chains them together. Composability without coupling.

Compound query: both planes in one conversation

Real workflow. Claude calls 3 MCP servers to build a complete picture.

  // Three data sources. One synthesized answer. Zero tab switches.
Build Your Own

Start with 3 Servers in 15 Minutes

You don't need 9 servers on day one. Start with the three that deliver the most value for SE work, then add more as you need them.

1

GitHub — Code access (2 minutes)

claude mcp add github --scope user -- npx -y @modelcontextprotocol/server-github

PR reviews, issue management, code search. Set GITHUB_PERSONAL_ACCESS_TOKEN in the env.

2

Context7 — Up-to-date docs (1 minute)

claude mcp add context7 --scope user https://mcp.context7.com/mcp

Current library docs for any framework. Free tier. No API key needed. Fixes the "stale training data" problem for technical work.

3

Slack — Communication (5 minutes)

claude mcp add slack --scope user -- npx -y @anthropic/slack-mcp

Search, read, send messages. Requires a Slack bot token with channels:history, chat:write, search:read scopes. Set SLACK_BOT_TOKEN.

+

Level Up: Build a Custom Server

Once you see the pattern, you can wrap any API. Our Meridian server is ~400 lines of Node.js. Each tool is a function that takes params, calls an API, and returns JSON. The MCP SDK handles the protocol.

If your team has internal APIs, dashboards, or data sources that Claude should access — a custom MCP server is 2-4 hours of work for a developer who's seen the pattern.
